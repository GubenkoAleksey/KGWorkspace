# AI Agent Directives & Project Architecture (FirestoreApp)

Цей файл містить ключові правила архітектури та написання коду для AI-асистента. AI повинен **завжди** звірятися з цим документом перед генерацією або рефакторингом коду.

## 🏗 Структура Проекту (Clean Architecture + Модульність)
Проект розділений на наступні модулі (застосовується жорстке розділення шарів):
- `:app` - Головний модуль, містить Navigation Graph, DI ініціалізацію та `Application` клас.
- `:core` - Спільний код: базові UI-компоненти (теми, типографіка, кнопки, діалоги), утиліти, загальні розширення.
- `:domain` - (Pure Kotlin) Бізнес-логіка, UseCases, Domain Models, Repository Interfaces. Не має жодних залежностей від Android фреймворку.
- `:data` - Реалізація репозиторіїв, робота з Firebase/Firestore, API клієнти, DTO, Room та мапери (з Data-моделей у Domain-моделі).
- `:feature:*` (`:feature:auth`, `:feature:home`, `:feature:status`) - UI екрани та ViewModels кожної окремої фічі.

## 🎨 UI Architecture & Composables Rules

### 1. Атомарна декомпозиція та структура
- **Принцип декомпозиції:** Кожен екран має бути розбитий на максимально прості, незалежні Composable-функції.
- **Семантичне найменування:** Використовуй описові назви за призначенням (наприклад, `OrderSummarySection`, а не `TopPart`).
- **🛑 AI Trigger (Ліміт файлу):** Довжина файлу НЕ повинна перевищувати 150–200 рядків. Якщо при генерації коду файл стає більшим — AI повинен автоматично виносити допоміжні Composable у пакет `components/` або окремі файли.

### 2. Ієрархія перевикористання та DRY
- **Пріоритет `:core` модуля:**
  - **🛑 AI Trigger (Пошук):** ПЕРЕД створенням нового базового UI-елемента (кнопки, інпути, картки, лоадери), AI спочатку повинен перевірити модуль `:core` на наявність готових.
- **Feature Level:** Шаблонні елементи для конкретної фічі виносяться у пакет `components` всередині модуля фічі (наприклад, `com.hubenko.feature.auth.ui.components`).
- **Заборона дублювання:** Жодна ідентична логіка відображення не повинна дублюватися. Завжди створюй перевикористовуваний компонент.

### 3. Стандарти розробки (Quality Assurance)
- **Stateless vs Stateful:** Чітко розділяй UI на два типи:
  - *Stateful (розумні):* Функції на рівні екрана. Працюють з ViewModel, збирають state (напр. `collectAsStateWithLifecycle`), передають дані та колбеки у Stateless-компоненти.
  - *Stateless (дурні):* Тільки приймають дані (`UI State` або примітиви) та івенти (`(Event) -> Unit`). **Пріоритет завжди віддається Stateless.**
- **Обов'язкові Previews:**
  - **🛑 AI Trigger (Генерація Preview):** Для кожного створеного Stateless Composable AI повинен генерувати `@Preview` з використанням теми проекту.
  - Для цілих екранів генеруй Previews для всіх станів: `Loading`, `Content`, `Error`.

## 🔄 Еволюція архітектури (Self-Update Rule)
- **🛑 AI Trigger (Зміна архітектури):** Якщо під час виконання завдань створюються нові модулі, змінюються архітектурні підходи або правила організації коду, AI **зобов'язаний** автоматично оновити цей файл (`AGENTS.md`), щоб він завжди відображав актуальний стан проекту.

## ✅ AI Agent Check-list (Алгоритм для AI перед генерацією коду)
1. [ ] Чи стосується запит архітектури? Якщо так, чи правильно я розміщую код (Domain -> Data -> Feature)?
2. [ ] Я перевірив модуль `:core` на наявність існуючих компонентів перед створенням нових UI-елементів?
3. [ ] Чи мій Composable `Stateless`? Якщо він `Stateful`, чи це дійсно кореневий компонент екрана?
4. [ ] Чи створив я `@Preview` для нових UI-елементів та перевірив стани екрана?
5. [ ] Чи файл, який я редагую або створюю, залишається в межах 150-200 рядків? (Якщо ні -> декомпозую).
6. [ ] Чи дотримуюсь я Unidirectional Data Flow (UDF) у взаємодії між UI та ViewModel?
7. [ ] Чи відбулися зміни в архітектурі, структурі модулів або базових правилах? Якщо так -> оновлюю `AGENTS.md`.
