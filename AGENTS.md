# AI Agent Directives & Project Architecture (FirestoreApp)

Цей файл містить ключові правила архітектури та написання коду для AI-асистента. AI повинен **завжди** звірятися з цим документом перед генерацією або рефакторингом коду.

## 🏗 Структура Проекту (Clean Architecture + MVI)
Проект базується на принципах Clean Architecture та патерні **MVI (Model-View-Intent)** з використанням **UDF (Unidirectional Data Flow)**.

### Розподіл по модулях:
- `:app` - Головний модуль, містить Navigation Graph, DI ініціалізацію та `Application` клас.
- `:core` - Спільний код: базові UI-компоненти (теми, типографіка, кнопки, діалоги), утиліти, загальні розширення.
- `:domain` - (Pure Kotlin) Бізнес-логіка, UseCases, Domain Models, Repository Interfaces.
- `:data` - Реалізація репозиторіїв, робота з Firebase/Firestore, DTO, Room та мапери.
- `:feature:*` - UI екрани та ViewModels. Кожна фіча реалізує MVI.

### Package Naming:
- Root package: `com.hubenko`
- App: `com.hubenko.firestoreapp`
- Core: `com.hubenko.core`
- Domain: `com.hubenko.domain`
- Data: `com.hubenko.data`
- Features: `com.hubenko.feature.{featurename}`

## 🎨 Правило Атомарної Декомпозиції та Візуальної Ізоляції

### 1. Принцип "Один файл — один головний компонент"
- **Уникати "файлів-смітників" (God-files):** Кожен логічно завершений UI-елемент (Header, Button, Card, Dialog) повинен мати власний файл у пакеті `components`. Це спрощує навігацію та зменшує конфлікти при злитті.
- **🛑 AI Trigger (Декомпозиція):** ПЕРЕД створенням компонента, якщо він логічно незалежний — створюй для нього новий файл. Не дописуй у існуючі файли з назвами типу `*Components.kt`.

### 2. Ієрархічна структура перевикористання
- **Global Components (`:core`):** Елементи, що використовуються у всьому додатку (напр., `AppTopBar`, `AppTextField`, `PrimaryActionButton`).
- **Feature-level Common:** Елементи, спільні лише для конкретної фічі (виносяться у пакет `ui.components` модуля фічі).
- **Atomic Part Decomposition:** Навіть у межах одного складного компонента/діалогу, виносити великі частини (Header, Content, Footer) в окремі файли, якщо структура стає складною.

### 3. Інкапсуляція стилів та "Базові компоненти"
- Замість копіювання параметрів (`colors`, `modifier`, `shape`) у кожен екземпляр Material-компонентів, створювати базовий компонент у `:core` (напр., `AppTextField`).
- Базовий компонент має задавати єдиний стиль проекту та приймати специфічний контент або параметри через лямбди.

### 4. Семантичне іменування
- Іменувати компоненти за їх **функціональною роллю**, а не за технічною реалізацією.
- Приклад: `AddEmployeeButton` замість `BlueBorderButton`. Це полегшує розуміння бізнес-логіки.

### 5. Обов'язкова наявність @Preview
- Кожен UI-компонент **МУСИТЬ** мати функцію превью.
- Превью мають відображати різні стани (Loading, Error, Empty, Content).
- Це дозволяє розробляти UI в ізоляції та слугує документацією.

## 🏛 Стандарти MVI та Stateless UI
- **Stateless vs Stateful:** 
  - *Stateful:* Підписується на `ViewState`, обробляє `SideEffect` та передає `Intent` у ViewModel.
  - *Stateless:* Тільки приймає `State` та лямбди для надсилання `Intent`.
- **Розділення Screen та Content (ОБОВ'ЯЗКОВО):** `*Screen.kt` (Stateful) та `*Content.kt` (Stateless) **завжди** зберігаються у різних файлах.
- **UDF (Unidirectional Data Flow):** Дані йдуть тільки вниз (State), івенти — тільки вгору (Intent).
- ViewModels наслідують `BaseViewModel<S, I, E>` з `:core` та використовують Hilt для ін'єкції залежностей.

## 🛠 Валідація та Надійність (Reliability)
- **🛑 AI Trigger (Валідація збірки):** Після внесення будь-яких змін у код, AI **зобов'язаний** переконатися, що проект успішно збирається через `.\gradlew.bat build`.
