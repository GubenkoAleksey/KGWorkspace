# AI Agent Directives & Project Architecture (FirestoreApp)

Цей файл містить ключові правила архітектури та написання коду для AI-асистента. AI повинен **завжди** звірятися з цим документом перед генерацією або рефакторингом коду.

## 🏗 Структура Проекту (Clean Architecture + MVI)
Проект базується на принципах Clean Architecture та патерні **MVI (Model-View-Intent)** з використанням **UDF (Unidirectional Data Flow)**.

### Розподіл по модулях:
- `:app` - Головний модуль, містить Navigation Graph, DI ініціалізацію та `Application` клас.
- `:core` - Спільний код: базові UI-компоненти (теми, типографіка, кнопки, діалоги), утиліти, загальні розширення.
- `:domain` - (Pure Kotlin) Бізнес-логіка, UseCases, Domain Models, Repository Interfaces. Не має жодних залежностей від Android фреймворку.
- `:data` - Реалізація репозиторіїв, робота з Firebase/Firestore, API клієнти, DTO, Room та мапери (з Data-моделей у Domain-моделі).
- `:feature:*` - UI екрани та ViewModels. Кожна фіча реалізує MVI через `ViewState`, `ViewIntent` та `ViewSideEffect`.

## 🎨 UI Architecture & Composables Rules

### 1. Атомарна декомпозиція та структура
- **Принцип декомпозиції:** Кожен екран має бути розбитий на максимально прості, незалежні Composable-функції.
- **Семантичне найменування:** Використовуй описові назви за призначенням (наприклад, `OrderSummarySection`, а не `TopPart`).
- **🛑 AI Trigger (Ліміт файлу):** Довжина файлу НЕ повинна перевищувати 150–200 рядків. Якщо при генерації коду файл стає більшим — AI повинен автоматично виносити допоміжні Composable у пакет `components/` або окремі файли.

### 2. Ієрархія перевикористання та DRY
- **Пріоритет `:core` модуля:** ПЕРЕД створенням нового базового UI-елемента, AI спочатку повинен перевірити модуль `:core` на наявність готових.
- **Feature Level:** Шаблонні елементи для конкретної фічі виносяться у пакет `components` всередині модуля фічі.

### 3. Стандарти MVI та Stateless UI
- **Stateless vs Stateful:** 
  - *Stateful:* Підписується на `ViewState`, обробляє `SideEffect` та передає `Intent` у ViewModel.
  - *Stateless:* Тільки приймає `State` та лямбди для надсилання `Intent`.
- **Розділення Screen та Content (ОБОВ'ЯЗКОВО):** Stateful-обгортка (наприклад, `HomeScreen.kt`, яка містить ViewModel) та її Stateless-контент (наприклад, `HomeContent.kt`, який приймає лише State та лямбди) **повинні зберігатися у різних файлах**.
- **UDF (Unidirectional Data Flow):** Дані йдуть тільки вниз (State), івенти — тільки вгору (Intent). Жодного прихованого стану всередині UI, якщо це не стосується анімацій або скролу.

## 🛠 Валідація та Надійність (Reliability)
- **🛑 AI Trigger (Валідація збірки):** Після внесення будь-яких змін у код, AI **зобов'язаний** переконатися, що проект успішно збирається. У разі виникнення помилок компіляції, AI повинен проаналізувати їх та виправити до завершення завдання.

## 🔄 Еволюція архітектури (Self-Update Rule)
- **🛑 AI Trigger (Зміна архітектури):** Якщо змінюються підходи або структура, AI **зобов'язаний** автоматично оновити цей файл.
